This guide covers common issues you may encounter when building with Chat and how to resolve them.

## Key Management Errors

### `UnlockFailed: InvalidPin`

**Symptom:** `unlock()` fails with an invalid PIN error.

**Causes:**
- Wrong PIN entered
- PIN has too many failed attempts (locked out)
- Keys were set up with a different PIN

**Solutions:**
```python
# The SDK raises a built-in ValueError; the message carries the reason.
try:
    chat.unlock(pin)
except ValueError as e:
    msg = str(e)
    if "InvalidPin" in msg:
        print("Wrong PIN. Check your PIN and try again.")
        # The message includes guesses_remaining when the realm reports it.
    elif "RateLimitExceeded" in msg or "Locked" in msg:
        print("Too many failed attempts. The keys are locked.")
```

<Note>
The Python binding raises the built-in `ValueError` (not a custom exception class) with a descriptive message. The JavaScript/TypeScript binding throws an `Error` whose message includes the Juicebox `reason` (e.g. `InvalidPin`, `RateLimitExceeded`, `NotRegistered`).
</Note>

<Warning>
After too many failed PIN attempts, Juicebox will permanently lock your keys. There is no recovery — you'll need to generate new keys and re-register them with X.
</Warning>

### `KeysNotUnlocked`

**Symptom:** Encryption/decryption operations fail because keys aren't loaded.

**Cause:** You're trying to encrypt or decrypt before calling `unlock()`.

**Solution:**
```python
# Always unlock before any crypto operations
if not chat.is_unlocked():
    chat.unlock(pin)

# Now you can encrypt/decrypt
encrypted = chat.encrypt_message(...)
```

### `KeyNotFound` / `ConversationKeyMissing`

**Symptom:** Can't decrypt a message because the conversation key is missing.

**Causes:**
- You don't have the conversation key for this message's `key_version`
- Key rotation happened and you missed the `KeyChange` event

**Solutions:**

1. **Fetch conversation events to get the current key events:**
```python
# get_conversation_events returns the key change events in meta.conversation_key_events.
key_events = []
for page in client.chat.get_conversation_events(conversation_id):
    key_events += (page.meta.conversation_key_events or []) if page.meta else []

# Extract and cache the raw conversation keys.
extracted = chat.extract_conversation_keys(key_events)
conversation_keys[conversation_id] = extracted["keys"]   # { version: key_bytes }
```

2. **Make sure you're handling key changes:**
```python
def handle_event(event_data):
    payload = event_data.get('data', {}).get('payload', {})
    key_change = payload.get('conversation_key_change_event')
    if key_change:
        # Extract and store the new key FIRST
        extracted = chat.extract_conversation_keys([key_change])
        conversation_keys[payload['conversation_id']] = extracted['keys']

    # Then decrypt the message (payload['encoded_event'])
    decrypt_message(event_data)
```

### `PublicKeyNotFound`

**Symptom:** Can't encrypt a message for a recipient because their public key isn't available.

**Solution:**
```python
# Fetch the recipient's public keys from the API.
resp = client.chat.get_user_public_keys(
    recipient_id,
    public_key_fields=["version", "public_key", "signing_public_key", "identity_public_key_signature"],
)
record = resp.data[0]

# Build the input for prepare_conversation_keys (identity key + version).
public_key_input = {
    "user_id": recipient_id,
    "public_key": record["public_key"],   # identity public key
    "key_version": record["version"],
}
```

## Decryption Errors

### `DecryptionFailed: InvalidCiphertext`

**Symptom:** Message decryption fails with invalid ciphertext error.

**Causes:**
- Message was corrupted in transit
- Wrong conversation key version used
- Message is malformed

**Solutions:**

1. **Verify you have the key version the event needs:**
```python
# The event payload carries the key version it was encrypted with.
message_key_version = payload.get('conversation_key_version')

# Make sure that version is in your cached keys for this conversation.
keys = conversation_keys.get(payload['conversation_id'], {})
if message_key_version not in keys:
    print(f"Missing key version {message_key_version}")
    # Re-extract from payload['conversation_key_change_event'] or fetch /events
```

2. **Check message integrity:**
```python
import base64

# Verify the encoded_event is valid base64
try:
    decoded = base64.b64decode(payload['encoded_event'])
except Exception:
    print("Invalid base64 encoding in encoded_event")
```

---

### `SignatureVerificationFailed`

**Symptom:** Message decrypts but signature verification fails.

**Causes:**
- Sender's signing key doesn't match
- Message was tampered with
- You have outdated signing keys for the sender

**Solutions:**
```python
# Fetch fresh public keys for the sender and build the signing-key entries.
resp = client.chat.get_user_public_keys(
    sender_id,
    public_key_fields=["version", "public_key", "signing_public_key", "identity_public_key_signature"],
)
signing_keys = [
    {
        "user_id": sender_id,
        "public_key_version": r["version"],
        "public_key": r["signing_public_key"],
        "identity_public_key": r["public_key"],
        "identity_public_key_signature": r["identity_public_key_signature"],
    }
    for r in resp.data
]

# Retry decryption with the full signing-key entries.
event = chat.decrypt_event(encoded_event, conversation_keys, signing_keys)

# Check verification status (dict field).
if not event["verified"]:
    print("Warning: Message signature could not be verified")
    # Decide whether to accept unverified messages
```

<Note>
Unverified messages aren't necessarily malicious — the sender may have rotated their signing key. Consider your threat model when deciding how to handle them.
</Note>

## Webhook Errors

### CRC Challenge Failing

**Symptom:** Webhook registration fails or webhook gets disabled.

**Causes:**
- Incorrect HMAC calculation
- Wrong consumer secret
- Response format incorrect

**Solution:**
```python
import hmac
import hashlib
import base64

def handle_crc(crc_token: str, consumer_secret: str) -> dict:
    # Must use HMAC-SHA256
    signature = hmac.new(
        consumer_secret.encode('utf-8'),
        crc_token.encode('utf-8'),
        hashlib.sha256
    ).digest()
    
    # Must be base64 encoded
    response_token = base64.b64encode(signature).decode('utf-8')
    
    # Must return exactly this format
    return {
        'response_token': f'sha256={response_token}'
    }
```

**Checklist:**
- [ ] Using the correct `consumer_secret` (API Secret Key, not API Key)
- [ ] Response is JSON with `response_token` field
- [ ] Token is prefixed with `sha256=`
- [ ] Responding within 3 seconds

### `403 Forbidden` on Webhook Events

**Symptom:** Your webhook receives events but returns 403.

**Cause:** Signature verification is failing.

**Solution:**
```python
def verify_webhook_signature(request) -> bool:
    signature = request.headers.get('x-twitter-webhooks-signature', '')
    payload = request.get_data()  # Raw bytes, not parsed JSON
    
    expected = 'sha256=' + base64.b64encode(
        hmac.new(
            CONSUMER_SECRET.encode('utf-8'),
            payload,  # Use raw payload, not request.json
            hashlib.sha256
        ).digest()
    ).decode('utf-8')
    
    # Use constant-time comparison
    return hmac.compare_digest(signature, expected)
```

<Warning>
**Common mistake:** Using `request.json` instead of `request.get_data()`. The signature is computed over the raw request body, not the parsed JSON.
</Warning>

### Not Receiving Events

**Symptom:** Webhook is registered but you're not getting chat events.

**Checklist:**

1. **Verify subscription exists:**
```bash
curl -X GET "https://api.x.com/2/activity/subscriptions" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

2. **Check event types:**
```bash
# Make sure you subscribed to both event types
# chat.received - messages TO you
# chat.sent - messages FROM you
```

3. **Verify user authorization:**
   - Chat events require OAuth 2.0 user authorization
   - User must have granted `dm.read` scope

4. **Check webhook health:**
```bash
# Trigger a manual CRC check
curl -X PUT "https://api.x.com/2/webhooks/WEBHOOK_ID" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

## API Errors

### `401 Unauthorized`

**Causes:**
- Expired access token
- Missing or invalid OAuth signature
- Wrong authentication method

**Solutions:**

```python
# For OAuth 2.0 - refresh the token
if response.status_code == 401:
    new_token = refresh_oauth2_token(refresh_token)
    # Retry request with new token
```

**Required scopes for Chat:**
- `dm.read` — Read conversations, events, and public keys
- `dm.write` — Send messages, initialize keys, create conversations, register public keys
- `tweet.read` — Required alongside `dm.*` on most chat endpoints
- `users.read` — Read user profiles and public keys
- `media.write` — Upload and download media attachments

---

### `429 Too Many Requests`

**Cause:** Rate limit exceeded.

**Solution:**
```python
import time

def make_request_with_retry(func, *args, max_retries=3):
    for attempt in range(max_retries):
        response = func(*args)
        
        if response.status_code == 429:
            # Get reset time from headers
            reset_time = int(response.headers.get('x-rate-limit-reset', 0))
            wait_seconds = max(reset_time - time.time(), 1)
            
            print(f"Rate limited. Waiting {wait_seconds}s...")
            time.sleep(wait_seconds)
            continue
        
        return response
    
    raise Exception("Max retries exceeded")
```

### `400 Bad Request` on Send Message

**Causes:**
- Invalid encrypted payload format
- Missing required fields
- Conversation key version mismatch

**Solution:**
```python
import uuid

# Generate the message_id yourself and pass the RAW conversation key.
message_id = str(uuid.uuid4())
payload = chat.encrypt_message(
    message_id,                 # message_id
    sender_id,                  # sender_id
    conversation_id,            # conversation_id
    raw_conversation_key,       # RAW bytes — not the encrypted base64 key
    text,                       # text
    conversation_key_version,   # conversation_key_version
    signing_key_version,        # signing_key_version
)

# Map SendPayload fields to the request body.
from xdk.chat.models import SendMessageRequest

client.chat.send_message(
    conversation_id,
    SendMessageRequest(
        message_id=message_id,
        encoded_message_create_event=payload.encrypted_content,
        encoded_message_event_signature=payload.encoded_event_signature,
    ),
)
```

<Warning>
Two common causes of `400` here: passing the **encrypted** conversation key instead of the raw bytes, and using the old `SendPayload` field names. The request body fields are `encoded_message_create_event` (= `payload.encrypted_content`) and `encoded_message_event_signature` (= `payload.encoded_event_signature`).
</Warning>

## Juicebox Errors

### `NetworkError` / `RealmUnreachable`

**Symptom:** Can't connect to Juicebox realms.

**Causes:**
- Network connectivity issues
- Realm is temporarily down
- Firewall blocking connections

**Solutions:**

1. **Check network connectivity:**
```python
import requests

def check_realm_connectivity(realm_url: str) -> bool:
    try:
        response = requests.get(f"{realm_url}/health", timeout=5)
        return response.status_code == 200
    except Exception as e:
        print(f"Realm unreachable: {e}")
        return False
```

2. **Implement retry logic:**
```python
import time

def unlock_with_retry(chat, pin, max_retries=3):
    for attempt in range(max_retries):
        try:
            chat.unlock(pin)
            return True
        except NetworkError:
            wait = 2 ** attempt  # Exponential backoff
            print(f"Network error, retrying in {wait}s...")
            time.sleep(wait)
    return False
```

### `TokenExpired`

**Symptom:** Juicebox operations fail with expired token error.

**Cause:** Auth tokens for realms have expired.

**Solution:**
```typescript
// TypeScript example with token refresh
const chat = await createChat({
  juiceboxConfig,
  getAuthToken: async (realmId) => {
    // This callback is called when tokens are needed
    // Return fresh tokens each time
    return await fetchFreshToken(realmId);
  }
});
```

## Debugging Tips

### Enable Debug Logging

```python
import logging

# Enable debug logs for Chat XDK
logging.getLogger('chat_xdk').setLevel(logging.DEBUG)

# Enable debug logs for HTTP requests
logging.getLogger('urllib3').setLevel(logging.DEBUG)
```

### Inspect Encrypted Payloads

```python
import base64
import json

def inspect_payload(encoded_event: str):
    """Decode and inspect an encoded event (without decrypting)."""
    try:
        decoded = base64.b64decode(encoded_event)
        print(f"Payload size: {len(decoded)} bytes")
        
        # Check if it looks like valid ciphertext
        if len(decoded) < 32:
            print("Warning: Payload seems too small")
        
        return decoded
    except Exception as e:
        print(f"Invalid base64: {e}")
        return None
```

### Verify Key Versions

```python
def verify_setup():
    """Verify encryption setup is correct."""
    print(f"Keys unlocked: {chat.is_unlocked()}")

    if chat.is_unlocked():
        # Local fingerprint of the loaded identity key (for out-of-band comparison).
        print(f"Local fingerprint: {chat.get_public_key_fingerprint()}")

        # Compare with what's registered on X.
        resp = client.chat.get_user_public_keys(
            my_user_id,
            public_key_fields=["version", "public_key", "signing_public_key"],
        )
        for key in resp.data:
            print(f"Registered version {key['version']}: identity={key['public_key'][:16]}...")
```
