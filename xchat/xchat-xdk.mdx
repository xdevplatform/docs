The **Chat XDK** handles key management, encryption, decryption, and signing for Chat. It does **not** call the X HTTP API—pair it with the [Python](/xdks/python/overview) or [TypeScript](/xdks/typescript/overview) **XDK**, or with HTTPS and a user access token.

App walkthrough: [Getting Started](/xchat/getting-started). Sample bots: [chat-xdk/examples](https://github.com/xdevplatform/chat-xdk/tree/main/examples).

### Install

| Language | Install / import |
|:---------|:-----------------|
| Python | `pip install chat-xdk` → `from chat_xdk import Chat` |
| TypeScript / JS | `npm install @xdevplatform/chat-xdk` |
| Rust | crates `chat_xdk` or `chat-xdk-core` |
| Go | `github.com/xdevplatform/chat-xdk/go/chatxdk` |
| C# | `ChatXdk` package (.NET 8+; native library on the loader path) |
| Java | `com.xdevplatform.chatxdk` (JNA; same native library as .NET) |

### Conventions

- Method names: `snake_case` (Python/Rust), `camelCase` (JS/Java), `PascalCase` (Go/C#)
- Conversation keys between SDK calls are **raw bytes** (or `Uint8Array` / `byte[]`), except **Go**, which often uses **base64 strings** on the FFI boundary
- Pass the **raw** conversation key into `encrypt_message`—not the ECIES-wrapped key from the API (Go: base64 of the raw key)
- Decrypted event types are PascalCase (`"Message"`) except in **JavaScript** (`"message"`, camelCase fields)
- Release native resources in Go (`Close()`), C# (`Dispose` / `using`), and Java (try-with-resources)

## Quick start

Decrypt a backlog, cache keys, decrypt one event, encrypt a reply. Wire the send body to [`POST …/messages`](/x-api/chat/send-chat-message) as in [Getting Started](/xchat/getting-started).

  <Tab title="Python">
    ```python
    from chat_xdk import Chat

    chat = Chat(juicebox_config_json)  # or Chat() + import_keys(blob)
    chat.unlock("YOUR_PIN")
    chat.set_key_version(signing_key_version)

    result = chat.decrypt_events(raw_events, signing_keys)
    for dm in result["messages"]:
        ev = dm["event"]
        if ev.get("type") == "Message":
            print(ev.get("sender_id"), ev.get("content", {}).get("text"))

    cached = result["conversation_keys"]["keys"]
    event = chat.decrypt_event(one_event_b64, cached, sender_signing_keys)

    raw_key = cached[result["conversation_keys"]["latest_version"]]
    payload = chat.encrypt_message(
        message_id, sender_id, conversation_id, raw_key, "Hi!",
        conversation_key_version, signing_key_version,
    )
    ```

## API surface

Method names below are **Python** / Rust style. Other languages expose the same operations with local naming (see conventions above).

### Lifecycle and keys

| Method | Purpose |
|:-------|:--------|
| `Chat(config_json?)` / `createChat` (JS) | Construct; optional Juicebox config JSON |
| `update_config` | Refresh Juicebox config |
| `set_reject_unverified` | Fail decrypt when signature cannot be verified |
| `set_key_version` | Your registered public-key **version** from the API (call after unlock/import) |
| `generate_keypairs` | New identity + signing keys; register publics via API |
| `setup` / `unlock` / `change_pin` / `delete` / `lock` | Juicebox PIN lifecycle (JS: async; some bindings pass config on unlock) |
| `is_unlocked` / `has_identity_key` | Key state |
| `get_public_keys` / `get_public_key_fingerprint` | Public material / fingerprint |
| `export_keys` / `import_keys` | Local private-key blob (handle as a secret) |

### Conversation keys

| Method | Purpose |
|:-------|:--------|
| `prepare_conversation_keys` | **Preferred** — generate key, wrap for each participant |
| `generate_conversation_key` | Raw 32-byte key only |
| `decrypt_conversation_key` | Unwrap one ECIES blob → raw key |
| `encrypt_conversation_key_for_recipients` | Wrap an existing key for many users |
| `extract_conversation_keys` | From key-change event blobs → `{ keys, latest_version }` |

When POSTing to the API, rename SDK `encrypted_key` → **`encrypted_conversation_key`**. Keep the **raw** `conversation_key` for `encrypt_message` / `encrypt_stream`.

### Decrypt

| Method | When |
|:-------|:-----|
| `decrypt_events` | History / backlog; extracts keys; collects per-event errors (does not throw per message) |
| `decrypt_event` | One live event with keys you already cached; throws/raises on failure |

Pass **signing keys** for verification (`SigningKeyEntry`: map API `version` → `public_key_version`, `signing_public_key` → `public_key`, `public_key` → `identity_public_key`, plus `identity_public_key_signature` and `user_id`). Omit or pass empty to skip verification.

### Encrypt and related

| Method | Purpose |
|:-------|:--------|
| `encrypt_message` | Text (+ optional entities/attachments, TTL, notify) |
| `encrypt_reply` / `encrypt_add_reaction` / `encrypt_remove_reaction` | Replies and reactions (`sequence_id` targets) |
| `encrypt` / `decrypt` | UTF-8 metadata (e.g. encrypted group name) with conversation key |
| `encrypt_stream` / `decrypt_stream` | Media bytes — see [Media](/xchat/media) |
| `sign` / `verify` / `verify_key_binding` | Low-level signing helpers |
| `sign_add_members` / `sign_key_change` | Group / rotation action signatures |

**SendPayload → send-message body:** `encrypted_content` → `encoded_message_create_event`, `encoded_event_signature` → `encoded_message_event_signature`, plus your `message_id`.

### Utilities

Module-level (or `ChatXdkUtilities` on C#/Java): `bytes_to_base64` / `base64_to_bytes`, hex helpers, `detect_mime_type`, `detect_image_dimensions`.

## Important types (conceptual)

- **SendPayload** — output of `encrypt_*` message helpers  
- **PublicKeyRegistrationPayload** — output of `generate_keypairs`; POST to public-keys API  
- **SigningKeyEntry** — input to decrypt for verification  
- **PreparedConversationKeys** — output of `prepare_conversation_keys`  
- **Event / Message content** — decrypt results; JS uses camelCase discriminators (`message`, `contentType`)

Full field lists: language stubs in the [chat-xdk repo](https://github.com/xdevplatform/chat-xdk) (`docs/API.md`, `*.pyi`, `index.d.ts`).

---

## Errors

Python typically raises **`ValueError`** with a descriptive message (e.g. invalid PIN). JS throws **`Error`**. Go returns `(value, error)`. Prefer **`decrypt_events`** for history so one bad event does not abort the batch.
