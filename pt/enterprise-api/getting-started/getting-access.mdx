O acesso Enterprise exige um processo de solicitação e onboarding. Sua equipe de conta dedicada ajudará a configurar as credenciais e seu acesso.

<Button href="/forms/enterprise-api-interest">Solicitar acesso Enterprise</Button>

---

## Etapa 1: Solicite o acesso Enterprise

<Steps>
  <Step title="Envie uma solicitação">
    [Preencha o formulário de interesse Enterprise](/forms/enterprise-api-interest) com detalhes sobre sua organização e caso de uso.
  </Step>
  <Step title="Discuta suas necessidades">
    Nossa equipe de vendas entrará em contato para entender seu volume de dados, requisitos de endpoint e necessidades de suporte.
  </Step>
  <Step title="Receba um pacote personalizado">
    Receba um plano sob medida com preços, limites de taxa e níveis de acesso projetados para o seu caso de uso.
  </Step>
</Steps>

---

## Etapa 2: Faça o onboarding

Assim que o seu plano Enterprise estiver pronto, seu gerente de conta dedicado ajudará a configurar:

<Steps>
  <Step title="Crie seu App de desenvolvedor">
    Configure seu App no [Developer Console](https://console.x.com) com acesso de nível Enterprise.
  </Step>
  <Step title="Configure seu acesso">
    Seu gerente de conta ajudará a configurar os limites de taxa, o acesso aos endpoints e quaisquer ajustes personalizados.
  </Step>
  <Step title="Gere as credenciais">
    Gere suas API keys e tokens para autenticação.
  </Step>
</Steps>

---

## Etapa 3: Salve suas credenciais

Você receberá várias credenciais dependendo das suas necessidades de autenticação:

| Credencial | Finalidade |
|:-----------|:--------|
| **API Key & Secret** | Identificam seu App. Usados para gerar tokens e assinar solicitações OAuth 1.0a. |
| **Bearer Token** | Autenticação apenas de App para ler dados públicos. |
| **Access Token & Secret** | Faça solicitações em nome da sua própria conta (OAuth 1.0a). |
| **Client ID & Secret** | Autenticação OAuth 2.0 para solicitações em contexto de usuário. |

<Warning>
**Salve imediatamente.** As credenciais são exibidas apenas uma vez. Armazene-as em um gerenciador de senhas ou cofre seguro. Se você as perder, precisará regenerá-las (o que invalida as anteriores).
</Warning>

## De quais credenciais você precisa?

<Tabs>
  <Tab title="Ler dados públicos">
    Use o **Bearer Token** para acesso simples e somente leitura a dados públicos.
    
    ```bash
    curl "https://api.x.com/2/users/by/username/xdevelopers" \
      -H "Authorization: Bearer $BEARER_TOKEN"
    ```
    
    Ideal para: pesquisar publicações, consultar usuários, ler tendências.
  </Tab>
  <Tab title="Agir como um usuário">
    Use **OAuth 2.0** (recomendado) ou **OAuth 1.0a** para agir em nome dos usuários.
    
    O OAuth 2.0 oferece scopes granulares — solicite apenas as permissões de que você precisa.
    
    Ideal para: publicar, curtir, seguir, acessar DMs.
    
    [Guia de OAuth 2.0 →](/resources/fundamentals/authentication/oauth-2-0/overview)
  </Tab>
  <Tab title="Agir como você mesmo">
    Use seu **Access Token & Secret** para fazer solicitações como sua própria conta.
    
    Esses tokens representam a conta que possui o App.
    
    Ideal para: testes, bots pessoais, sua própria automação.
  </Tab>
</Tabs>

## Boas práticas de segurança para credenciais

<CardGroup cols={2}>
  <Card title="Use variáveis de ambiente" icon="terminal">
    Nunca coloque credenciais diretamente no código-fonte.
  </Card>
  <Card title="Não faça commit no git" icon="code-branch">
    Adicione arquivos de credenciais ao `.gitignore`.
  </Card>
  <Card title="Faça rotação regularmente" icon="arrows-rotate">
    Regenere as credenciais periodicamente como medida de segurança.
  </Card>
  <Card title="Use scopes mínimos" icon="shield-check">
    Solicite apenas as permissões OAuth de que seu App precisa.
  </Card>
</CardGroup>
