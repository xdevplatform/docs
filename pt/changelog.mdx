<Update label="May 4, 2026" description="X API v2">
  ### Migração dos endpoints de busca para o novo índice de busca

  Hoje, migramos os endpoints de busca da X API v2 (`/2/tweets/search/recent`, `/2/tweets/search/all`, `/2/tweets/counts/recent`, `/2/tweets/counts/all`) para o novo índice de busca da plataforma principal X. Isso traz grandes melhorias de desempenho, redução drástica de erros 503 e três novos operadores de precisão: `min\_likes:`, `min\_replies:` e `min\_reposts:`. Observação: reposts não são mais retornados em resultados de busca baseados em palavras-chave (o Filtered Stream não foi alterado). Saiba mais [aqui](https://devcommunity.x.com/t/important-update-x-api-search-endpoints-migrating-to-new-search-index/264595).
</Update>

<Update label="Apr 16, 2026" description="X API">
  ### Atualização de preços da X API: Owned Reads agora $0,001

  A partir de 20 de abril de 2026, introduzimos novos preços de "Owned Reads" a $0,001 por recurso (1.000 por $1) para endpoints qualificados ao acessar seus próprios dados: `GET /2/users/{id}/bookmarks`, `/blocking`, `/muting`, `/pinned_lists`, `/tweets`, `/mentions`, `/liked_tweets`, `/followers`, `/following`, `/owned_lists`, `/followed_lists` e `/list_memberships`. Também atualizamos as gravações — `POST /2/tweets` agora custa $0,015 por publicação, e Posts contendo uma URL custam $0,20 por Post (respostas invocadas permanecem $0,01). Following, Likes e Quote-Posts via API foram removidos de todos os níveis de autoatendimento. Saiba mais [aqui](https://devcommunity.x.com/t/x-api-pricing-update-owned-reads-now-0-001-other-changes-effective-april-20-2026/263025).
</Update>

  Hoje, anunciamos a descontinuação de `POST /2/account_activity/replay/webhooks/{webhook_id}/subscriptions/all`, efetiva em 25 de março de 2026 às 12:00 PM ET. Use o novo endpoint consolidado `POST /2/webhooks/replay` em vez disso — ele fornece funcionalidade idêntica. Saiba mais [aqui](https://devcommunity.x.com/t/deprecation-of-account-activity-replay-endpoint/260354).

<Update label="Mar 18, 2026" description="X API v2">
  ### Filtro de direção para assinaturas da X Activity API

  Hoje, adicionamos um filtro opcional `direction` às assinaturas da X Activity API. Os assinantes agora podem delimitar eventos DM e de chat para `sent`, `received` ou ambos, reduzindo o ruído quando apenas eventos de saída ou entrada são necessários.
</Update>

<Update label="Mar 13, 2026" description="X API v2">
  ### Atualizações no formato de tipos de eventos na X Activity API

  Hoje, anunciamos mudanças na nomenclatura dos tipos de eventos na X Activity API, efetivas em 18 de março de 2026. Os tipos de eventos em PascalCase (por exemplo, `ProfileBioUpdate`) estão sendo descontinuados em favor de um formato em minúsculas separado por pontos (por exemplo, `profile.update.bio`). Atualize qualquer filtragem, instruções switch ou lógica de desserialização que dependa dos valores antigos. Saiba mais [aqui](https://devcommunity.x.com/t/updates-to-event-types-format-in-the-x-activity-api/259765).
</Update>

<Update label="Mar 11, 2026" description="X API v2">

</Update>

<Update label="Feb 23, 2026" description="X API v2">
  ### Combatendo o spam gerado por LLM

  Hoje, fizemos alterações para reduzir respostas automatizadas e de baixa qualidade no X. Respostas programáticas via `POST /2/tweets` agora são permitidas apenas quando o autor do Post original "invocou" o respondedor (mencionando essa conta com @ ou citando um de seus Posts). Restrições adicionais se aplicam à @menção ou citação programática de usuários. Essas alterações afetam apenas os níveis de autoatendimento — o acesso Enterprise não é impactado. Saiba mais [aqui](https://devcommunity.x.com/t/x-api-v2-update-addressing-llm-generated-spam/257909).
</Update>

<Update label="Sep 19, 2025" description="X API v2">
  ### Connections API para gerenciar conexões de streaming ativas

  Hoje, lançamos a [Connections API](https://docs.x.com/x-api/connections/introduction) para gerenciar conexões de streaming ativas programaticamente:
  - `GET /2/connections` — lista conexões de streaming ativas e históricas
  - `DELETE /2/connections/all` — encerra todas as conexões de streaming ativas do app
  - `DELETE /2/connections/{endpoint_id}` — encerra conexões para um endpoint de streaming específico
  - `DELETE /2/connections` — encerra conexões específicas por UUID

  Isso ajuda os desenvolvedores a resolver erros "TooManyConnections" e limpar conexões obsoletas sem esperar timeouts.
</Update>

<Update label="Sep 9, 2025" description="X API v2">
  ### Lançamento da Filtered Stream Webhooks API

  Hoje, lançamos a [Filtered Stream Webhooks API](https://docs.x.com/x-api/webhooks/stream/introduction), que entrega Posts públicos correspondentes assincronamente a um webhook registrado em vez de exigir uma conexão persistente. As regras são gerenciadas usando os endpoints existentes `/2/tweets/search/stream/rules`, e Posts correspondentes (incluindo histórico de edições) são entregues como JSON ao seu webhook. Disponível para desenvolvedores Enterprise; suporta mais de 25.000 regras por projeto com regras de até 2.048 caracteres. Saiba mais [aqui](https://devcommunity.x.com/t/announcing-the-filtered-stream-webhooks-api/248464).

<Update label="Jul 8, 2025" description="X API v2">

  Hoje, lançamos a [Community Notes API](https://docs.x.com/x-api/community-notes/introduction) — um novo conjunto de endpoints para AI Note Writers participarem programaticamente do Community Notes:

### Alterações na renderização do payload de Quote
Hoje, estamos adicionando uma alteração na forma como as URLs serão renderizadas no payload de quote.

  Estamos adicionando uma nova entidade chamada "quoted_status_permalink" para garantir que o post citado possa ser referenciado e removeremos o link t.co do campo "text" do post citado.

  Com o novo formato, o link t.co não é mais anexado ao campo "text" do post citado no objeto "quoted_status".

  Além disso, não colocaremos mais a url t.co, expanded_url e display_url do post citado na entidade 'urls'. Em vez disso, estes serão adicionados a um novo objeto "quoted_status_permalink" com atributos "url", "expanded" e "display" URL. Isso significa que o objeto "quoted_status_permalink" será reservado para o link t.co de volta ao post citado, enquanto o array "entities.urls" pode ser usado para identificar quaisquer links compartilhados dentro do post ou quote original.

  Por favor, leia nosso [post do fórum](https://devcommunity.x.com/t/updating-how-urls-are-rendered-in-the-quote-tweet-payload/105473) e [documentação atualizada](https://devcommunity.x.com/t/upcoming-changes-to-the-developer-platform/104603) para mais detalhes.
</Update>
