## Ciclo de vida y claves

Construye el SDK, guarda las claves privadas (copia de seguridad segura de claves protegida con cĂłdigo de acceso o un blob de claves local), registra las claves **pĂşblicas** con la Chat API y establece tu **versiĂłn de clave pĂşblica** registrada despuĂ©s de unlock o import. La copia de seguridad segura de claves estĂˇ implementada con **Juicebox**, y por eso los campos de configuraciĂłn relacionados llevan ese nombre. Llama a `generate_keypairs` una vez por identidad de dispositivo/app; envĂ­a el payload de registro con POST al endpoint de claves pĂşblicas. Usa `setup` / `unlock` (y helpers de cĂłdigo de acceso relacionados) para la copia de seguridad segura de claves en cada binding. `export_keys` / `import_keys` (persistencia de blob de clave en bruto para bots y servidores) estĂˇn disponibles solo en los **bindings nativos**â€”Python, Go, .NET, JVM y Rust. El binding JS/WASM no expone la exportaciĂłn ni importaciĂłn de claves en bruto: en un navegador cualquier script que acceda a la instancia podrĂ­a exfiltrar la identidad, asĂ­ que JS mantiene las claves dentro de la copia de seguridad segura de claves. Un servidor JS que quiera evitar un round-trip al realm de copia de seguridad por solicitud deberĂ­a reutilizar una Ăşnica instancia `Chat` desbloqueada entre solicitudes, o ejecutar un binding nativo donde se admitan blobs de clave.

## Claves de conversaciĂłn

Tres mĂ©todos **prepare** hacen que una sola llamada haga todo lo que un cambio de clave necesita: generar una nueva clave de conversaciĂłn, cifrarla para cada participante (a partir de las claves pĂşblicas que pases) y firmar el cambio. Todos devuelven la misma forma **`PreparedConversationChange`**, lista para POSTâ€”renombra el campo del SDK `encrypted_key` a **`encrypted_conversation_key`** en `conversation_participant_keys` y mapea las firmas de acciĂłn al campo de cuerpo requerido **`action_signatures`**.

| Escenario | MĂ©todo | Firmas de acciĂłn devueltas |
|:----------|:-------|:---------------------------|
| Iniciar un 1:1 (omite el id de conversaciĂłnâ€”el SDK lo deriva) o rotar la clave de cualquier conversaciĂłn (pasa el id) | `prepare_conversation_key_change` | 1 |
| Crear un grupo (id acuĂ±ado por `POST /2/chat/conversations/group/initialize`) | `prepare_group_create` | 2â€”envĂ­a ambas |
| Agregar miembros a un grupo | `prepare_group_members_change` | 2â€”envĂ­a ambas |

Conserva los bytes de la clave **en bruto** para `encrypt_message` y multimedia; nunca pases el sobre cifrado de la API a encrypt.

<Warning>
**Verifica las claves obtenidas antes de envolverlas.** Los mĂ©todos prepare cifran la nueva clave de conversaciĂłn hacia cualesquiera claves pĂşblicas que pases. Antes de pasarlas, llama a `verify_key_binding(identity, signing, signature)` en cada registro obtenidoâ€”sus campos `public_key`, `signing_public_key` e `identity_public_key_signature` de la API de claves pĂşblicasâ€”para que una clave de identidad sustituida no pueda recibir la clave de conversaciĂłn.
</Warning>

Usa `extract_conversation_keys` en los payloads de eventos de cambio de clave para reconstruir `{ keys, latest_version }`. `decrypt_conversation_key` desenvuelve un solo blob ECIES.

## Descifrar

**`decrypt_events`** es para el historial y el backlog: extrae las claves de conversaciĂłn del stream, devuelve los mensajes descifrados y **recolecta** errores por evento en lugar de fallar el lote completo. **`decrypt_event`** es para un solo evento en vivo cuando ya tienes un cachĂ© de claves; lanza excepciĂłn/throw ante un fallo.

Pasa **claves de firma** para que el SDK pueda verificar a los remitentes. Mapea los campos de clave pĂşblica de la API a `SigningKeyEntry`: `public_key_version` â†’ `public_key_version` (mismo nombre), `signing_public_key` â†’ `public_key`, `public_key` â†’ `identity_public_key`, mĂˇs `identity_public_key_signature` y `user_id`. La verificaciĂłn es obligatoria por defecto: omitir o pasar una lista vacĂ­a de claves de firma **no** la saltaâ€”los eventos firmados fallan (recolectados en `errors` para `decrypt_events`, lanzados para `decrypt_event`). Para realmente saltar la verificaciĂłn primero debes llamar a `set_reject_unverified(false)` (no recomendado en producciĂłn).

## Helpers de cifrado y envĂ­o

**`encrypt_message`** construye el texto cifrado firmado para un mensaje de texto (entidades opcionales, adjuntos vĂ­a `media_hash_key`, TTL, flags de notificaciĂłn). Mapea el payload devuelto al cuerpo send-message: `encrypted_content` â†’ **`encoded_message_create_event`**, `encoded_event_signature` â†’ **`encoded_message_event_signature`**, mĂˇs tu **`message_id`**.

Usa **`encrypt_reply`**, **`encrypt_add_reaction`** y **`encrypt_remove_reaction`** para respuestas y reacciones (`sequence_id` apunta al padre). **`encrypt` / `decrypt`** son para metadatos UTF-8 bajo la clave de conversaciĂłn (por ejemplo, un nombre de grupo cifrado)â€”no sobres de mensajes. **`encrypt_stream` / `decrypt_stream`** cifran los bytes de los adjuntos; consulta [Multimedia](/xchat/media). Los mĂ©todos de bajo nivel **`sign` / `verify` / `verify_key_binding`** admiten flujos avanzados; los cambios de clave de conversaciĂłn, creaciones de grupo y adiciones de miembros son firmados por los [mĂ©todos prepare](#claves-de-conversaci-n).

El id de conversaciĂłn pasado a `encrypt_message` / `encrypt_reply` puede ser cualquier forma que tengasâ€”`A:B` de eventos, `A-B` de listados o rutas URL (en cualquier orden) o solo el id de usuario del destinatarioâ€”el SDK lo canonicaliza antes de firmar. Los ids de grupo (con prefijo `g`) pasan sin cambios.

### Streaming incremental para multimedia grande

Para archivos grandes, evita mantener el payload completo en memoria: `stream_encryptor()` / `stream_decryptor()` devuelven un `StreamEncryptor` / `StreamDecryptor` al que alimentas por chunks (unos 1 MB cada uno) con `push(chunk)`, y luego llamas a `finish()` una vez al final. Al descifrar, `finish()` detecta un stream truncado (falla si la entrada terminĂł antes del frame final), asĂ­ que no trates el texto plano acumulado como completo hasta que tenga Ă©xito.

<Warning>
**Solo JS/WASM:** `finish()` consume y libera el objeto WASM subyacenteâ€”nunca llames a `free()` despuĂ©s de `finish()` (lanza error). Llama a `free()` solo para abandonar un stream *antes* de finalizar (por ejemplo, en una ruta de error).
</Warning>

<Tabs>
  <Tab title="Python">
    ```python
    enc = chat.stream_encryptor(raw_conversation_key)
    chunks = [enc.push(chunk) for chunk in read_in_chunks(file_bytes, 1 << 20)]
    chunks.append(enc.finish())
    ciphertext = b"".join(chunks)

    dec = chat.stream_decryptor(raw_conversation_key)
    out = [dec.push(chunk) for chunk in read_in_chunks(ciphertext, 1 << 20)]
    out.append(dec.finish())  # raises on truncation
    plain = b"".join(out)
    ```
  </Tab>
  <Tab title="TypeScript">
    ```typescript
    const enc = chat.streamEncryptor(rawConversationKey);
    const parts: Uint8Array[] = [];
    try {
      for (const chunk of readInChunks(fileBytes, 1 << 20)) parts.push(enc.push(chunk));
      parts.push(enc.finish()); // consumes + frees enc â€” do not call enc.free() after this
    } catch (e) {
      enc.free(); // only when abandoning before finish()
      throw e;
    }
    const ciphertext = concat(parts);
    ```

## Tipos importantes

Estos tipos conceptuales aparecen en varios lenguajes (los nombres exactos de los campos difieren; JS suele usar discriminadores de evento en camelCase como `message`):

- **SendPayload** â€” valor de retorno de `encrypt_message` y helpers de cifrado relacionados; mapea al cuerpo de envĂ­o de la Chat API.
- **PublicKeyRegistrationPayload** â€” salida de `generate_keypairs` / getters de claves pĂşblicas para la API add-public-key.
- **SigningKeyEntry** â€” material pĂşblico del remitente pasado a decrypt para la verificaciĂłn de firma.
- **PreparedConversationChange** â€” salida de los tres mĂ©todos prepare: el `conversation_id` derivado o pasado, los bytes de `conversation_key` en bruto, `conversation_key_version`, `participant_keys` (`user_id`, `encrypted_key`, `public_key_version`) y `action_signatures` (`message_id`, `encoded_message_event_detail`, `signature`, `signature_version`, `public_key_version`, `signature_payload` opcionalâ€”omitido en firmas de cambio de clave porque ese payload incrusta la clave en texto plano).
- **DecryptEventsResult** â€” mensajes, errores opcionales y `conversation_keys` extraĂ­das.

Para listas completas de campos, usa los stubs de lenguaje en el [repo chat-xdk](https://github.com/xdevplatform/chat-xdk) (`docs/API.md`, `*.pyi`, `index.d.ts`).

## Errores

Python normalmente lanza **`ValueError`** con un mensaje descriptivo (por ejemplo, un cĂłdigo de acceso invĂˇlido). TypeScript/JavaScript lanza **`Error`**. Go devuelve `(value, error)`. Prefiere **`decrypt_events`** para el historial de modo que un evento defectuoso no aborte el lote; inspecciona la colecciĂłn de errores para ver fallos parciales.

Algunos errores de verificaciĂłn son **permanentes**. Las firmas son inmutables y se verifican reconstruyendo el payload firmado desde el evento en sĂ­, asĂ­ que un evento antiguo que falla con `signature missing or no matching signing key` o un desajuste ECDSA fallarĂˇ en cada carga futuraâ€”ninguna reintentaciĂłn, actualizaciĂłn de claves o llamada a la API puede sanarlo. TrĂˇtalos como tombstones, no como errores transitorios. Rotar la clave de conversaciĂłn inicia un historial limpio y verificable desde ese punto en adelante.
