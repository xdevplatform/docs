Call [`GET /2/region`](/livestream-api/sources/get-recommended-region) (and follow the 307 redirect) for the current recommendation for your location. The final response contains the region code to use.

## Supported region codes

These region codes have been observed in use (the set can change over time):

| Region Code | Area |
|:------------|:-----|
| `us-east-1` | US East (N. Virginia) |
| `us-east-2` | US East (Ohio) |
| `us-west-1` | US West (N. California) |
| `us-west-2` | US West (Oregon) |
| `ca-central-1` | Canada (Central) |
| `eu-central-1` | Europe (Frankfurt) |
| `eu-west-1` | Europe (Ireland) |
| `eu-west-3` | Europe (Paris) |
| `ap-south-1` | Asia Pacific (Mumbai) |
| `ap-northeast-1` | Asia Pacific (Tokyo) |
| `ap-northeast-2` | Asia Pacific (Seoul) |
| `ap-southeast-1` | Asia Pacific (Singapore) |
| `ap-southeast-2` | Asia Pacific (Sydney) |
| `sa-east-1` | South America (São Paulo) |

<Warning>
**Do not derive or hardcode ingest hostnames from the region code.** The RTMP host serving a region is not predictable from its name — for example, `us-west-1` ingest is served from an Oregon host (`or.pscp.tv`), not a "us-west" host. Always use the `rtmp_url` / `rtmps_url` returned in the source object. They are of the form `rtmp://<host>:80/x` (insecure) and `rtmps://<host>:443/x` (secure — use this in production); the `x` path is standard for these ingest points.
</Warning>
